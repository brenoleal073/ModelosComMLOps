Relatório de Engenharia de Machine Learning

Projeto: Previsão de Cancelamento de Reservas Hoteleiras (MLOps Pipeline)

1. Estrutura do Projeto e Contexto de Negócio

Foco: Integração com o negócio, organização de código e planejamento orientado à entrega.

Contexto e Objetivos

O cancelamento de reservas é um dos principais desafios financeiros da indústria hoteleira. O objetivo deste projeto é desenvolver uma solução de Machine Learning capaz de prever o risco de cancelamento no momento da reserva, permitindo que o hotel adote estratégias de Revenue Management, como overbooking controlado ou campanhas de retenção.

Decisão de Engenharia: Política vs. Mecanismo

O projeto foi estruturado utilizando o padrão de design que separa a Política (regras de negócio e hiperparâmetros definidos em arquivos .yaml) do Mecanismo (scripts Python reutilizáveis). Esta abordagem reduz a dependência de notebooks e garante que o fluxo seja automatizado e independente de ambiente, utilizando a classe centralizadora PipelineContext.

2. Pipelines de Dados e Prevenção de Overfitting

Foco: Qualidade de dados, prevenção de Data Leakage e redução de dimensionalidade (PCA).

Prevenção de Data Leakage

Um erro comum neste dataset é a presença de colunas como reservation_status. No script preprocessamento.py, realizamos a remoção cirúrgica de variáveis que ocorrem após o fato (o cancelamento), garantindo que o modelo aprenda apenas com informações disponíveis no momento da reserva.

Pipeline de Features e Redução de Dimensionalidade

Implementamos um ColumnTransformer robusto que integra:

  • Imputação: Tratamento automático de nulos via mediana (numéricos) e constantes (categóricos).
  • Codificação: One-Hot Encoding com tratamento de categorias desconhecidas em produção.
  • PCA: Implementamos a redução de dimensionalidade opcional via configuração. Observamos que, embora o PCA reduza o custo computacional, ele impactou negativamente o Recall, o que justificou a manutenção do modelo Baseline para a versão final de produção.

3. Experimentação Rastreável com MLflow

Foco: Reprodutibilidade, comparação de modelos e ajuste de hiperparâmetros.

Comparação Sistemática

Utilizamos o MLflow para registrar todas as métricas (Precision, Recall, F1, AUC). Foram comparadas três abordagens principais:

Experimento F1-Score ROC-AUC Justificativa
Random Forest (Baseline) 0.812 0.935 Sólido, mas com parâmetros padrão.
Random Forest + PCA 0.763 0.904 Perda de performance significativa.
RF Tuned (Final) 0.831 0.945 Melhor equilíbrio via RandomizedSearchCV.

O ajuste de hiperparâmetros via RandomizedSearchCV com Validação Cruzada (3-folds) garantiu a generalização do modelo, mitigando o risco de overfitting detectado em execuções preliminares.

4. Operacionalização e Ciclo de Vida

Foco: Persistência, inferência funcional (FastAPI) e monitoramento de Drift.

Arquitetura de Serving

Em vez de utilizar o servidor padrão do MLflow, optamos pelo FastAPI por ser uma solução assíncrona de alta performance. A API foi desenhada para buscar o modelo "Campeão" diretamente do banco de dados do MLflow, garantindo que o deploy seja desacoplado do treinamento.

Visão de Ciclo de Vida

Para a operação contínua, o relatório propõe:

  • Detecção de Drift: Monitoramento das distribuições de lead_time (Data Drift) e da relação entre depósitos e cancelamentos (Concept Drift).
  • Estratégia de Retreino: Gatilhos automáticos baseados na degradação da métrica F1-Score em dados reais coletados via logs de inferência.

Conclusão

O projeto demonstra a transição de um modelo estatístico isolado para um sistema de engenharia robusto. A solução é rastreável, versionada e pronta para escala, cumprindo integralmente os requisitos de maturidade técnica e responsabilidade profissional exigidos pela disciplina.
